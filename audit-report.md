# Project Audit Report: parser-2gis

## Executive Summary

Проведён полный аудит проекта parser-2gis — парсера сайта 2GIS для сбора данных об организациях. Проект представляет собой сложное Python-приложение с поддержкой CLI и TUI интерфейсов, параллельным парсингом, кэшированием и множеством опций конфигурации.

**Общее состояние проекта: ХОРОШЕЕ**

Проект демонстрирует высокий уровень разработки с множеством реализованных лучших практик, включая:
- Типизацию через type hints и TypedDict
- Подробную документацию (docstrings)
- Обработку исключений
- Безопасную работу с ресурсами
- Оптимизации производительности

Однако выявлен ряд проблем различной критичности, требующих внимания.

## Critical Issues (Must Fix)

### 1. Потенциальная уязвимость SQL Injection в cache.py
**Файл:** `parser_2gis/cache.py`  
**Строка:** ~720-750 (метод `clear_batch`)  
**Проблема:** Несмотря на наличие валидации хешей `_validate_hash()`, метод использует динамическую генерацию SQL запроса с подстановкой плейсхолдеров. При определённых условиях возможна SQL injection атака.

**Код с проблемой:**
```python
# Строка ~720
placeholders = ', '.join('?' for _ in url_hashes)
sql = f"DELETE FROM cache WHERE url_hash IN ({placeholders})"
cursor.execute(sql, url_hashes)
```

**Почему это проблема:**
- Динамическая генерация SQL даже с плейсхолдерами потенциально опасна
- Если валидация `_validate_hash()` будет обойдена или изменена, возможна инъекция
- Лучшая практика — использовать только параметризованные запросы без динамической генерации

**Решение:** Использовать только строго параметризованные запросы, рассмотреть возможность использования ORM.

### 2. Жёсткое кодирование путей и конфигураций
**Файл:** `parser_2gis/paths.py`  
**Строка:** 14-15  
**Проблема:** Использование переменных окружения без валидации и fallback

**Код с проблемой:**
```python
if "_MEIPASS2" in os.environ:
    here = os.environ["_MEIPASS2"]
```

**Почему это проблема:**
- Переменные окружения могут быть подделаны
- Отсутствует валидация пути
- Возможна атака через подделку пути выполнения

## High Priority Issues

### 3. subprocess.Popen без полной валидации команды
**Файл:** `parser_2gis/chrome/browser.py`  
**Строка:** 126-133  
**Проблема:** Запуск subprocess.Popen с командой, содержащей пути из конфигурации

**Код с проблемой:**
```python
self._proc = subprocess.Popen(
    self._chrome_cmd,
    shell=False,
    stderr=subprocess.DEVNULL,
    stdout=subprocess.DEVNULL,
)
```

**Почему это проблема:**
- `chrome_options.binary_path` проходит валидацию, но путь может быть символической ссылкой
- Атака через symlink возможна если файл будет заменён после валидации
- **Хорошо:** `shell=False` предотвращает injection через оболочку

**Решение:** Добавить дополнительную валидацию через `os.path.realpath()` и проверку владельца файла.

### 4. Обработка signal handlers с потенциальной рекурсией
**Файл:** `parser_2gis/main.py`  
**Строка:** 230-270  
**Проблема:** Несмотря на флаг `_is_cleaning_up`, возможна рекурсия при частых сигналах

**Код с проблемой:**
```python
def _signal_handler(signum: int, frame: Any) -> None:
    global _interrupted, _is_cleaning_up
    
    if _is_cleaning_up:
        logger.warning("Получен повторный сигнал %d во время очистки...", signum)
        return
    
    _is_cleaning_up = True
    # ... очистка ...
```

**Почему это проблема:**
- Между проверкой флага и установкой возможна гонка
- В многопоточной среде возможна рекурсия

**Решение:** Использовать `threading.Lock()` для атомарности операций.

### 5. Утечка файловых дескрипторов при ошибке в parallel_parser.py
**Файл:** `parser_2gis/parallel_parser.py`  
**Строка:** 300-350  
**Проблема:** Цикл создания уникальных имён файлов может привести к утечке FD

**Код с проблемой:**
```python
for attempt in range(MAX_UNIQUE_NAME_ATTEMPTS):
    try:
        temp_fd = os.open(str(temp_filepath), os.O_CREAT | os.O_EXCL | os.O_WRONLY, mode=0o644)
        os.close(temp_fd)
        temp_fd = None
        break
    except FileExistsError:
        # Генерация нового имени, но FD может остаться открытым при OSError
```

**Почему это проблема:**
- При `OSError` в цикле файловый дескриптор может не закрыться
- В многопоточной среде возможна утечка нескольких FD

**Решение:** Использовать контекстный менеджер или гарантировать закрытие в `finally`.

## Medium Priority Issues

### 6. Избыточная сложность merge операции в parallel_parser.py
**Файл:** `parser_2gis/parallel_parser.py`  
**Строка:** 500-850 (метод `merge_csv_files`)  
**Проблема:** Метод содержит 350+ строк кода с множественными вложенными блоками try-finally

**Почему это проблема:**
- Высокая цикломатическая сложность
- Трудно тестировать
- Сложно поддерживать
- Множественные точки выхода

**Решение:** Рефакторинг — разбить на меньшие методы:
- `_acquire_merge_lock()`
- `_merge_csv_files()`
- `_delete_source_files()`
- `_release_merge_lock()`

### 7. Глобальное состояние в common.py
**Файл:** `parser_2gis/common.py`  
**Строка:** 50-90  
**Проблема:** Глобальные множества и паттерны для чувствительных данных

**Код с проблемой:**
```python
_SENSITIVE_KEYS: Set[str] = {...}
_SENSITIVE_KEY_PATTERN = re.compile(...)
```

**Почему это проблема:**
- Глобальное состояние усложняет тестирование
- Паттерны могут быть изменены в runtime
- Потенциальная гонка данных в многопоточной среде

**Решение:** Использовать module-level константы с `typing.Final` или вынести в отдельный конфиг.

### 8. Отсутствие лимита на размер загружаемых данных
**Файл:** `parser_2gis/parser/parsers/main.py`  
**Строка:** ~100-200  
**Проблема:** Парсер не ограничивает размер загружаемых данных от сервера

**Почему это проблема:**
- Возможна DoS атака через большой ответ сервера
- Потребление памяти не контролируется
- Риск падения приложения

**Решение:** Добавить проверку размера ответа и лимит памяти.

### 9. Жёстко закодированные таймауты
**Файл:** `parser_2gis/parser/parsers/main.py`  
**Строка:** 35-50  
**Проблема:** Константы таймаутов жёстко закодированы в модуле

```python
MAX_RESPONSE_ATTEMPTS: int = 3
NAVIGATION_TIMEOUT: int = 120
WAIT_REQUESTS_TIMEOUT: int = 120
```

**Почему это проблема:**
- Невозможно настроить без изменения кода
- Разные сетевые условия требуют разных таймаутов
- Усложняет тестирование

**Решение:** Вынести в `ParserOptions` как конфигурируемые параметры.

### 10. Недостаточная обработка ошибок в writer
**Файл:** `parser_2gis/writer/writers/csv_writer.py`  
**Строка:** ~180-200  
**Проблема:** Метод `_writerow` логирует ошибку но продолжает работу

```python
def _writerow(self, row: Dict[str, Any]) -> None:
    try:
        self._writer.writerow(row)
    except Exception as e:
        logger.error("Ошибка во время записи строки: %s", e)
```

**Почему это проблема:**
- Данные могут быть потеряны без уведомления
- Нет retry логики
- Нет fallback механизма

**Решение:** Добавить retry логику и счётчик ошибок с порогом остановки.

## Detailed Findings

### Code Quality

#### Положительные аспекты:
1. **Type hints:** Проект активно использует type hints, TypedDict, type aliases
2. **Docstrings:** Отличная документация с примерами использования
3. **Константы:** Магические числа вынесены в именованные константы
4. **Структура:** Чёткое разделение на модули (parser, writer, chrome, logger)
5. **Контекстные менеджеры:** Правильное использование `with` для ресурсов

#### Проблемные области:
1. **Цикломатическая сложность:** Некоторые методы превышают 50+ строк
2. **Глобальное состояние:** Несколько модулей используют глобальные переменные
3. **Длинные методы:** `merge_csv_files()` — 350+ строк, `parse_single_url()` — 200+ строк
4. **Повторение кода:** Некоторые паттерны повторяются в разных файлах

### Security

#### Реализованные меры безопасности:
1. **DNS rebinding защита** в `_validate_url()` (main.py)
2. **Валидация binary_path** с проверкой прав (browser.py)
3. **SQL injection prevention** через валидацию хешей (cache.py)
4. **Блокировка eval/exec** в JavaScript коде (remote.py)
5. **Очистка чувствительных данных** в логировании (common.py)
6. **TemporaryDirectory** для автоматической очистки профилей

#### Выявленные уязвимости:
1. **SQL Injection риск** в динамической генерации SQL (cache.py)
2. **Symlink атака** на путь браузера (browser.py)
3. **Переменные окружения** без валидации (paths.py)
4. **Отсутствие лимита памяти** для загружаемых данных

### Performance

#### Реализованные оптимизации:
1. **lru_cache** для кэширования результатов (common.py, remote.py)
2. **Connection pooling** для SQLite (cache.py)
3. **WAL режим** для SQLite (cache.py)
4. **Пакетные операции** для вставки/удаления (cache.py)
5. **Увеличенные буферы** для чтения/записи файлов (128KB)
6. **Компилированные regex** паттерны (multiple files)
7. **orjson** для быстрой JSON сериализации (cache.py)

#### Возможности улучшения:
1. **Асинхронность:** Проект использует потоки, можно рассмотреть asyncio
2. **Ленивая загрузка:** Некоторые данные загружаются сразу при импорте
3. **Генераторы:** Некоторые функции могут возвращать генераторы вместо списков

### Documentation

#### Сильные стороны:
1. **Полные docstrings** с секциями Args, Returns, Raises
2. **Примеры использования** в документации
3. **Комментарии на русском** языке
4. **README.md** с подробной инструкцией
5. **CHANGELOG.md** с историей изменений
6. **SECURITY.md** с политикой безопасности

#### Пробелы:
1. **Отсутствуют диаграммы** архитектуры
2. **Нет документации API** для разработчиков
3. **Мало примеров** для сложных сценариев использования

3. **Исправить гонку в signal handlers** с использованием Lock

6. **Исправить утечку FD** в parallel_parser.py
7. **Вынести таймауты** в конфигурацию ParserOptions

### Средний приоритет (1 месяц):
8. **Добавить retry логику** в writer с порогом остановки
9. **Улучшить покрытие тестами** writer модуля
10. **Добавить type hints** для всех функций
11. **Снизить глобальное состояние** через dependency injection

### Низкий приоритет (по возможности):
