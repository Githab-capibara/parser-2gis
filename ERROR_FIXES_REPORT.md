Отчёт об ошибках
===============

Проблема 1: Валидация JavaScript кода
--------------------------------------

Проблема:
Валидатор скриптов блокировал выполнение легитимного JavaScript кода с ошибкой:
"Обнаружена подозрительная конкатенация строк с function"

Где: parser_2gis/chrome/js_executor.py, функция _check_concatenation_bypass, строки ~198-220

Причина:
Функция _check_concatenation_bypass использовала чрезмерно строгую проверку:
- Удаляла все не-буквенные символы из кода
- Искала опасные слова (eval, function, setTimeout и др.) в оставшейся строке
- Это блокировало легитимный код, где эти слова встречаются как часть обычных строк

Например, код `"hello" + "world"` мог быть заблокирован, если в коде где-то было слово
"function" в другом контексте.

Решение:
Переписана функция _check_concatenation_bypass для проверки именно подозрительных паттернов:
- Теперь проверяется конкатенация строк вида `"ev" + "al"` или `"func" + "tion"`
- Используется regex паттерн для поиска разбитых опасных слов
- Добавлена проверка: опасное слово должно быть полным результатом конкатенации
- Легитимный код со строками больше не блокируется

Изменения в файле parser_2gis/chrome/js_executor.py:
- Функция _check_concatenation_bypass полностью переписана
- Добавлен regex паттерн для поиска конкатенации строк
- Добавлена логика проверки именно подозрительных паттернов обхода


Проблема 2: monitor_tab получает None от HTTP запроса
-----------------------------------------------------

Проблема:
AttributeError: 'NoneType' object has no attribute 'json'

Где: parser_2gis/chrome/remote.py, метод monitor_tab, строка ~545

Причина:
Функция _safe_external_request может вернуть None при ошибке HTTP запроса, но код
сразу вызывал ret.json() без проверки на None. Это происходило в цикле мониторинга
вкладки, когда DevTools endpoint временно недоступен.

Решение:
Добавлена проверка на None перед вызовом ret.json():
- Если ret is None, логгируется сообщение об ошибке
- Устанавливаются флаги tab_detached и _stopped для остановки вкладки
- Цикл мониторинга корректно завершается через break

Изменения в файле parser_2gis/chrome/remote.py:
- Строка ~545: добавлена проверка `if ret is None`
- Добавлено логгирование через app_logger.debug
- Добавлено корректное завершение цикла мониторинга


Проблема 3: Порты не освобождаются при запуске Chrome
-----------------------------------------------------

Проблема:
Chrome не может запуститься, потому что порты заняты от предыдущих запусков.

Где: parser_2gis/chrome/utils.py (функция free_port), parser_2gis/chrome/remote.py (метод start)

Причины:
1. Функция free_port() выбирала порт, но не гарантировала его освобождение перед использованием
2. Между выбором порта и запуском Chrome проходило время, порт мог быть занят
3. Недостаточное количество попыток ожидания запуска Chrome (всего 3 попытки)
4. Недостаточное время ожидания между попытками (максимум 3 секунды)
5. Кэш проверки портов не очищался при повторных попытках

Решение:
Внесены изменения в два файла:

1. parser_2gis/chrome/utils.py (функция free_port):
   - Добавлена опция SO_REUSEADDR для сокета (предотвращает проблемы с занятостью)
   - Добавлена небольшая задержка (0.01 сек) после закрытия сокета
   - Добавлен импорт модуля time

2. parser_2gis/chrome/remote.py (метод start):
   - Увеличено количество попыток: с 3 до 5
   - Увеличено максимальное время ожидания: с 3.0 до 5.0 секунд
   - Увеличено количество проверок порта: с 1 до 3 retries
   - Увеличен таймаут проверки порта: с 1.0 до 2.0 секунд
   - Добавлена очистка кэша портов (_clear_port_cache) перед финальной проверкой
   - Добавлена дополнительная проверка порта после исчерпания попыток


Профилактика
============

Рекомендации для предотвращения подобных ошибок в будущем:

1. Валидация JavaScript кода:
   - Тестировать валидатор на реальных примерах легитимного кода
   - Использовать whitelist подход (разрешать известное безопасное) вместо blacklist
   - Добавить юнит-тесты для функции _check_concatenation_bypass

2. Обработка HTTP запросов:
   - Всегда проверять возвращаемое значение на None перед использованием
   - Использовать try-except блоки для всех внешних запросов
   - Добавить логгирование для отладки временных проблем с сетью

3. Работа с портами:
   - Использовать SO_REUSEADDR для всех сокетов
   - Добавлять задержку после освобождения порта
   - Увеличивать количество попыток для операций с внешними сервисами
   - Очищать кэш при повторных попытках
   - Добавить мониторинг占用 портов для отладки

4. Общие рекомендации:
   - Добавить интеграционные тесты для запуска Chrome
   - Логгировать все этапы инициализации браузера
   - Добавить метрики времени запуска для мониторинга производительности
