# Releases

Этот документ описывает процесс релиза и версионирования проекта parser-2gis.

## Версионирование

Проект использует [Semantic Versioning](https://semver.org/lang/ru/):

```
MAJOR.MINOR.PATCH
```

- **MAJOR** — несовместимые изменения API
- **MINOR** — новая функциональность с обратной совместимостью
- **PATCH** — обратная совместимость и исправления ошибок

## Процесс релиза

### Подготовка релиза

1. **Обновление версии**
   ```bash
   # Обновите версию в parser_2gis/version.py
   version = "X.Y.Z"
   ```

2. **Обновление CHANGELOG.md**
   - Добавьте новую секцию с версией и датой
   - Опишите все значимые изменения
   - Укажите метрики архитектуры

3. **Обновление документации**
   - Проверьте актуальность README.md
   - Обновите ARCHITECTURE.md при необходимости
   - Проверьте CLI примеры

4. **Финальное тестирование**
   ```bash
   # Запуск всех тестов
   pytest tests/ -v --cov=parser_2gis
   
   # Проверка линтеров
   ruff check .
   black --check .
   
   # Проверка типов
   mypy parser_2gis/
   ```

### Публикация релиза

1. **Создание тега Git**
   ```bash
   git add .
   git commit -m "Release v X.Y.Z"
   git tag -a "vX.Y.Z" -m "Release version X.Y.Z"
   git push origin main --tags
   ```

2. **Публикация на PyPI**
   ```bash
   # Сборка пакета
   python -m build
   
   # Публикация на PyPI
   python -m twine upload dist/*
   ```

3. **GitHub Release**
   - Создайте релиз на GitHub
   - Добавьте описание из CHANGELOG.md
   - Прикрепите бинарные файлы (опционально)

### Пост-релиз

1. **Обновление документации**
   - Обновите документацию на сайте (если есть)
   - Обновите примеры использования

2. **Коммуникация**
   - Уведомите пользователей о релизе
   - Обновите социализацию проекта

## Критерии релиза

### Major релиз (X.0.0)
- Несовместимые изменения API
- Значительные архитектурные изменения
- Удаление deprecated функциональности

### Minor релиз (0.Y.0)
- Новая функциональность
- Улучшения производительности
- Обратная совместимость сохранена

### Patch релиз (0.0.Z)
- Исправления ошибок
- Улучшения безопасности
- Небольшие улучшения

## Экстренные исправления (Hotfix)

Для критических исправлений используется процедура hotfix:

1. Создайте ветку от тега релиза
   ```bash
   git checkout -b hotfix/X.Y.Z+1 vX.Y.Z
   ```

2. Внесите исправления

3. Обновите PATCH версию

4. Следуйте стандартному процессу релиза

5. Сделайте merge в main

## Deprecation Policy

Функциональность помечается как deprecated:

1. **Предупреждение** — добавляется DeprecationWarning
2. **Документирование** — указывается в CHANGELOG.md
3. **Период** — минимум 1 minor релиз для миграции
4. **Удаление** — в следующем major релизе
