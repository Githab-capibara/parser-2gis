# Project Analyzer

Скилл для комплексного анализа проекта, исправления ошибок, оптимизации кода и синхронизации с GitHub.

## Рабочий процесс

### Шаг 1: Анализ структуры проекта

**ПЕРВОЙ КОМАНДОЙ** всегда выполняйте `tree` для получения структуры проекта:

```bash
tree -L 3 -I '__pycache__|*.pyc|.git'
```

Это покажет все файлы и директории проекта. Запомните структуру для дальнейшего анализа.

**Важно**: На каждую команду устанавливайте таймаут 5 минут (300000 мс).

### Шаг 2: Чтение всех файлов проекта

Прочитайте все файлы проекта для полного понимания контекста:

1. Используйте `read_file` для каждого файла кода
2. Используйте `glob` для поиска всех файлов определённого типа
3. Используйте `grep_search` для поиска конкретных паттернов

**Приоритет файлов для чтения**:
- Конфигурационные файлы (pyproject.toml, setup.cfg, package.json, etc.)
- Основные файлы кода (.py, .js, .ts, etc.)
- Тестовые файлы
- Документация (README.md, docs/)

### Шаг 3: Анализ кода на ошибки

Проверьте код по следующим критериям:

**Логические ошибки**:
- Неправильные условия в if/else
- Ошибки в циклах (off-by-one, бесконечные циклы)
- Неправильная обработка граничных случаев
- Ошибки в вычислениях

**Синтаксические ошибки**:
- Нарушение синтаксиса языка
- Неправильное использование скобок
- Ошибки импорта

**Ошибки стиля**:
- Нарушение PEP 8 (для Python) или аналогичных стандартов
- Непоследовательное именование
- Отсутствие отступов

**Уязвимости безопасности**:
- Хардкод секретов (пароли, токены)
- SQL инъекции
- XSS уязвимости

### Шаг 4: Исправление ошибок и оптимизация

**Правила исправления**:

1. **Сохраняйте исходную структуру** - не меняйте архитектуру без необходимости
2. **Сохраняйте функциональность** - код должен делать то же самое
3. **Оптимизируйте синтаксис** - делайте код более читаемым
4. **Улучшайте производительность** - устраняйте узкие места

**Примеры оптимизаций**:
- Замена циклов на list comprehensions (где уместно)
- Устранение дублирования кода
- Выделение повторяющейся логики в функции
- Упрощение сложных условий

### Шаг 5: Перевод комментариев на русский

**Все комментарии** должны быть на русском языке:

1. Переведите все существующие комментарии
2. Сохраняйте технические термины на английском (функции, классы, переменные)
3. Добавляйте комментарии только там, где это необходимо для понимания

**Пример**:
```python
# Было:
# Calculate the sum of all elements
total = sum(elements)

# Стало:
# Вычислить сумму всех элементов
total = sum(elements)
```

### Шаг 6: Синхронизация с GitHub

**После всех изменений** синхронизируйте код с GitHub:

1. Проверьте статус изменений:
   ```bash
   git status
   ```

2. Просмотрите изменения:
   ```bash
   git diff HEAD
   ```

3. Добавьте файлы:
   ```bash
   git add .
   ```

4. Сделайте коммит с осмысленным сообщением:
   ```bash
   git commit -m "Исправление: [краткое описание]"
   ```

5. Отправьте изменения:
   ```bash
   git push
   ```

**Или используйте GitHub MCP инструменты**:
- `mcp__github__create_or_update_file` - для обновления файлов
- `mcp__github__push_files` - для пуша нескольких файлов
- `mcp__github__create_pull_request` - для создания PR

## Правила

### Таймауты

**НА КАЖДУЮ КОМАНДУ** устанавливайте таймаут 5 минут (300000 мс):

```json
{
  "timeout": 300000
}
```

Это предотвратит зависание долгих операций.

### Язык комментариев

- **Все комментарии** - только на русском
- **Технические термины** - на английском (имена функций, классов, переменных)
- **Документация** - на русском, с примерами кода

### Работа с GitHub

- **Никогда** не пушьте файл QWEN.md
- **Всегда** проверяйте изменения перед коммитом
- **Используйте** осмысленные сообщения коммитов
- **Синхронизируйте** изменения после каждой значимой правки

## Вспомогательные скрипты

Скилл использует следующие скрипты:

- `scripts/timeout_command.py` - выполнение команд с таймаутом
- `scripts/github_sync.py` - синхронизация с GitHub
- `scripts/code_analyzer.py` - статический анализ кода

## Лучшие практики

1. **Сначала план** - всегда создавайте план перед действиями
2. **Проверка контекста** - читайте файлы перед редактированием
3. **Минимальные изменения** - меняйте только то, что необходимо
4. **Тестирование** - проверяйте, что код работает после изменений
5. **Документирование** - описывайте значимые изменения
