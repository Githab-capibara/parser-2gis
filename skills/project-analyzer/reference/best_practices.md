## 2. Поиск ошибок

### 2.1. Логические ошибки

**Что искать:**
- Неправильные условия в `if/else`
- Ошибки в циклах (off-by-one)
- Отсутствие обработки граничных случаев
- Неправильный порядок операций
- Потенциальные null/None значения

**Примеры:**
```python
# ❌ Ошибка: off-by-one
for i in range(len(items) - 1):  # Пропускает последний элемент
    process(items[i])

# ✅ Правильно:
for i in range(len(items)):
    process(items[i])

# Или лучше:
for item in items:
    process(item)
```

```python
# ❌ Ошибка: неправильное условие
if status == "active" or "running":  # Всегда True
    handle_active()

# ✅ Правильно:
if status in ("active", "running"):
    handle_active()
```

### 2.2. Синтаксические ошибки

**Что искать:**
- Несбалансированные скобки
- Отсутствие двоеточий
- Неправильные отступы
- Ошибки импорта

### 2.3. Уязвимости безопасности

**Что искать:**
- Хардкод секретов (пароли, токены, ключи)
- Использование `eval()` / `exec()`
- SQL инъекции
- Командные инъекции
- Небезопасная десериализация

**Примеры:**
```python
# ❌ Опасно: хардкод пароля
DB_PASSWORD = "super_secret_123"

# ✅ Правильно: использование переменных окружения
import os
DB_PASSWORD = os.getenv('DB_PASSWORD')
```

```python
# ❌ Опасно: SQL инъекция
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")

# ✅ Правильно: параметризованный запрос
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))
```

## 3. Оптимизация кода

### 3.1. Производительность

**Паттерны оптимизации:**

1. **Замена циклов на comprehensions:**
```python
# ❌ Медленно:
result = []
for i in range(100):
    if i % 2 == 0:
        result.append(i * 2)

# ✅ Быстрее:
result = [i * 2 for i in range(100) if i % 2 == 0]
```

2. **Использование генераторов:**
```python
# ❌ Создаёт список в памяти:
squares = [x**2 for x in range(1000000)]

# ✅ Генератор (ленивое вычисление):
squares = (x**2 for x in range(1000000))
```

3. **Кэширование:**
```python
# ❌ Повторяющиеся вычисления:
def fibonacci(n):
    if n < 2:
        return n
    return fibonacci(n-1) + fibonacci(n-2)

# ✅ С кэшированием:
from functools import lru_cache

@lru_cache(maxsize=None)
def fibonacci(n):
    if n < 2:
        return n
    return fibonacci(n-1) + fibonacci(n-2)
```

### 3.2. Читаемость

**Правила:**
- Функции должны быть короткими (< 50 строк)
- Одна функция = одна ответственность
- Используйте понятные имена
- Избегайте магических чисел

```python
# ❌ Плохо:
def calc(a, b, c):
    return a * b * c * 1.2

# ✅ Хорошо:
def calculate_total_price(price, quantity, tax_rate=0.2):
    """Вычислить итоговую стоимость с налогом."""
    return price * quantity * (1 + tax_rate)
```

### 3.3. Устранение дублирования

**Правило трёх повторений:** если код повторяется 3 раза — выделите в функцию.

```python
# ❌ Дублирование:
def send_email(to, subject, body):
    # ... отправка email
    log_message(f"Email sent to {to}")
    update_stats("email_sent")
    notify_admin("email sent")

def send_sms(to, message):
    # ... отправка SMS
    log_message(f"SMS sent to {to}")
    update_stats("sms_sent")
    notify_admin("sms sent")

# ✅ Без дублирования:
def send_notification(channel, to, content):
    """Отправить уведомление и залогировать."""
    # ... отправка
    log_message(f"{channel} sent to {to}")
    update_stats(f"{channel}_sent")
    notify_admin(f"{channel} sent")
```

## 4. Перевод комментариев

### 4.1. Правила перевода

1. **Технические термины** остаются на английском:
   - Функции, классы, переменные
   - Названия библиотек и фреймворков
   - Протоколы и стандарты

2. **Объяснения** переводятся на русский:
   - Описание логики
   - Пояснения к алгоритмам
   - Предупреждения и заметки

### 4.2. Примеры перевода

```python
# ❌ Было (на английском):
# Initialize the database connection pool
# Set maximum retries to 3
# Calculate the total amount with tax

# ✅ Стало (на русском):
# Инициализировать пул соединений с базой данных
# Установить максимальное количество попыток равным 3
# Вычислить итоговую сумму с налогом
```

```python
# ❌ Было:
# FIXME: This is a temporary hack
# TODO: Implement proper error handling
# HACK: Workaround for the API bug

# ✅ Стало:
# FIXME: Это временное решение
# TODO: Реализовать правильную обработку ошибок
# HACK: Обход ошибки API
```

### 4.3. Форматы комментариев

Используйте стандартные маркеры:

- `# FIXME:` — известная проблема, требующая исправления
- `# TODO:` — функциональность для будущей реализации
- `# NOTE:` — важное замечание
- `# WARNING:` — предупреждение о потенциальных проблемах
- `# XXX:` — опасный код, требующий осторожности

## 5. Работа с GitHub

### 5.1. Перед коммитом

```bash
# Проверить статус
git status

# Просмотреть изменения
git diff HEAD

# Убедиться, что QWEN.md не включён
git ls-files | grep QWEN.md
```

### 5.2. Сообщение коммита

**Плохо:**
```
fix bug
update
changes
```

**Хорошо:**
```
fix: исправление ошибки деления на ноль в модуле calculator

- Добавлена проверка делителя перед вычислением
- Возврат None при делении на ноль
- Добавлен юнит-тест для граничного случая
```

### 5.3. Частота коммитов

- Делайте коммиты после каждой логической единицы изменений
- Не смешивайте разные типы изменений в одном коммите
- Коммитьте часто — это упрощает откат и ревью
