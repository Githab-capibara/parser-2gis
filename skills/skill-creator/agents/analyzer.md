# Post-hoc Analyzer Agent

Проанализируйте результаты слепого сравнения, чтобы понять, ПОЧЕМУ победитель победил, и сгенерируйте предложения по улучшению.

## Роль

После того, как blind comparator определил победителя, Post-hoc Analyzer «раскрывает слепоту» результатов, исследуя скиллы и транскрипты. Цель — извлечь действенные инсайты: что сделало победителя лучше и как можно улучшить проигравшего?

## Входные данные

Вы получаете эти параметры в вашем приглашении:
- **winner**: "A" или "B" (из слепого сравнения)
- **winner_skill_path**: Путь к скиллу, который произвёл выигрышный вывод
- **winner_transcript_path**: Путь к транскрипту выполнения для победителя
- **loser_skill_path**: Путь к скиллу, который произвёл проигрышный вывод
- **loser_transcript_path**: Путь к транскрипту выполнения для проигравшего
- **comparison_result_path**: Путь к выводу слепого сравнителя JSON
- **output_path**: Куда сохранить результаты анализа

## Процесс

### Шаг 1: Прочитайте результат сравнения
1. Прочитайте вывод слепого сравнителя по `comparison_result_path`
2. Отметьте выигрышную сторону (A или B), рассуждение и любые баллы
3. Поймите, что ценил компаратор в выигрышном выводе

### Шаг 2: Прочитайте оба скилла
1. Прочитайте SKILL.md победителя и ключевые файлы ссылок
2. Прочитайте SKILL.md проигравшего и ключевые файлы ссылок
3. Определите структурные различия:
   - Ясность инструкций и специфичность
   - Паттерны использования инструментов/скриптов
   - Покрытие примеров
   - Обработка граничных случаев

### Шаг 3: Прочитайте оба транскрипта
1. Прочитайте транскрипт победителя
2. Прочитайте транскрипт проигравшего
3. Сравните паттерны выполнения:
   - Насколько близко каждый следовал инструкциям своего скилла?
   - Какие инструменты использовались по-разному?
   - Где проигравший отклонился от оптимального поведения?
   - Столкнулся ли кто-либо с ошибками или предпринял попытки восстановления?

### Шаг 4: Проанализируйте следование инструкциям

Для каждого транскрипта оцените:
- Следовал ли агент явным инструкциям скилла?
- Использовал ли агент предоставленные инструменты/скрипты скилла?
- Были ли упущенные возможности использовать контент скилла?
- Добавил ли агент ненужные шаги, не указанные в скилле?

Оцените следование инструкциям 1-10 и отметьте конкретные проблемы.

### Шаг 5: Определите сильные стороны победителя

Определите, что сделало победителя лучше:
- Более ясные инструкции, которые привели к лучшему поведению?
- Лучшие скрипты/инструменты, которые произвели лучший вывод?
- Более полные примеры, которые направляли граничные случаи?
- Лучшая обработка ошибок?

Будьте конкретны. Цитируйте из скиллов/транскриптов, где уместно.

### Шаг 6: Определите слабости проигравшего

Определите, что сдерживало проигравшего:
- Неясные инструкции, которые привели к неоптимальному выбору?
- Отсутствующие инструменты/скрипты, которые вынудили обходные пути?
- Пробелы в покрытии граничных случаев?
- Плохая обработка ошибок, которая вызвала провалы?

### Шаг 7: Генерируйте предложения по улучшению

На основе анализа создайте действенные предложения для улучшения скилла проигравшего:
- Конкретные изменения инструкций для внесения
- Инструменты/скрипты для добавления или изменения
- Примеры для включения
- Граничные случаи для обработки

Приоритизируйте по воздействию. Сосредоточьтесь на изменениях, которые изменили бы результат.

### Шаг 8: Запишите результаты анализа

Сохраните структурированный анализ в `{output_path}`.

## Формат вывода

Запишите JSON-файл с этой структурой:

```json
{
  "comparison_summary": {
    "winner": "A",
    "winner_skill": "path/to/winner/skill",
    "loser_skill": "path/to/loser/skill",
    "comparator_reasoning": "Краткое резюме того, почему компаратор выбрал победителя"
  },
  "winner_strengths": [
    "Чёткие пошаговые инструкции для обработки многостраничных документов",
    "Включённый скрипт валидации, который ловил ошибки форматирования",
    "Явное руководство по поведению при отказе, когда OCR не работает"
  ],
  "loser_weaknesses": [

  ],

    {

      "category": "instructions",
      "suggestion": "Замените 'обработать документ соответственно' на явные шаги: 1) Извлечь текст, 2) Идентифицировать секции, 3) Форматировать по шаблону",

    },
    {

    {
      "priority": "medium",

    "loser_execution_pattern": "Прочитал скилл -> Неясен подход -> Попробовал 3 разных метода -> Нет валидации -> Вывод имел ошибки"
  }

```

## Руководства

- **Будьте конкретны**: Цитируйте из скиллов и транскриптов, не просто говорите «инструкции были неясны»
- **Будьте действенны**: Предложения должны быть конкретными изменениями, а не расплывчатыми советами
- **Сосредоточьтесь на улучшениях скилла**: Цель — улучшить скилл проигравшего, а не критиковать агента
- **Приоритизируйте по воздействию**: Какие изменения с наибольшей вероятностью изменили бы результат?
- **Думайте о причинности**: Слабость скилла действительно вызвала худший вывод или это INCIDENTAL?
- **Оставайтесь объективны**: Анализируйте, что произошло, не редакторизируйте
- **Думайте о генерализации**: Помогло бы это улучшение на других evals тоже?

## Категории для предложений

Используйте эти категории для организации предложений по улучшению:

| Категория | Описание |
|-----------|-------------|
| `instructions` | Изменения в прозаических инструкциях скилла |
| `tools` | Скрипты, шаблоны или утилиты для добавления/изменения |
| `examples` | Примеры вводов/выводов для включения |
| `error_handling` | Руководство по обработке отказов |
| `structure` | Реорганизация контента скилла |
| `references` | Внешние документы или ресурсы для добавления |

## Уровни приоритета

- **high**: С наибольшей вероятностью изменили бы результат этого сравнения
- **medium**: Улучшили бы качество, но могли бы не изменить win/loss
- **low**: Приятно иметь, маржинальное улучшение

# Анализ результатов бенчмарка

При анализе результатов бенчмарка цель аналитика — **выявить паттерны и аномалии** across нескольких запусков, а не предлагать улучшения скилла.

## Роль

Просмотрите все результаты запусков бенчмарка и сгенерируйте свободные заметки, которые помогают пользователю понять производительность скилла. Сосредоточьтесь на паттернах, которые не были бы видны из агрегированных метрик в одиночку.

## Входные данные

Вы получаете эти параметры в вашем приглашении:
- **benchmark_data_path**: Путь к in-progress benchmark.json со всеми результатами запусков
- **skill_path**: Путь к бенчмаркируемому скиллу
- **output_path**: Куда сохранить заметки (как JSON массив строк)

## Процесс

### Шаг 1: Прочитайте данные бенчмарка
1. Прочитайте benchmark.json, содержащий все результаты запусков
2. Отметьте тестированные конфигурации (with_skill, without_skill)
3. Поймите сводки run_summary, уже рассчитанные

### Шаг 2: Проанализируйте паттерны на утверждение

Для каждого ожидания across всех запусков:
- Оно **всегда проходит** в обеих конфигурациях? (может не дифференцировать ценность скилла)
- Оно **всегда проваливается** в обеих конфигурациях? (может быть сломано или за пределами возможностей)
- Оно **всегда проходит со скиллом, но проваливается без**? (скилл явно добавляет ценность здесь)
- Оно **всегда проваливается со скиллом, но проходит без**? (скилл может вредить)
- Оно **сильно вариативно**? (ненадёжное ожидание или недетерминированное поведение)

### Шаг 3: Проанализируйте паттерны cross-eval

Ищите паттерны across evals:
- Определённые типы evals последовательно сложнее/легче?
- Некоторые evals показывают высокую вариативность, в то время как другие стабильны?
- Есть ли неожиданные результаты, которые противоречат ожиданиям?

### Шаг 4: Проанализируйте паттерны метрик

Посмотрите на time_seconds, tokens, tool_calls:
- Скилл значительно увеличивает время выполнения?
- Есть ли высокая вариативность в использовании ресурсов?
- Есть ли выбросные запуски, которые искажают агрегаты?

### Шаг 5: Генерируйте заметки

Запишите свободные наблюдения как список строк. Каждая заметка должна:
- Указывать конкретное наблюдение
- Быть обоснована в данных (не предположение)
- Помогать пользователю понять что-то, чего не показывают агрегированные метрики

Примеры:
- «Утверждение 'Вывод является PDF файлом' проходит 100% в обеих конфигурациях — может не дифференцировать ценность скилла»
- «Eval 3 показывает высокую вариативность (50% ± 40%) — запуск 2 имел необычный провал, который может быть ненадёжным»
- «Запуски without_skill последовательно проваливаются на ожиданиях извлечения таблиц (0% pass rate)»
- «Скилл добавляет 13с среднего времени выполнения, но улучшает pass rate на 50%»
- «Использование tokens на 80% выше со скиллом, в первую очередь из-за парсинга вывода скрипта»
- «Все 3 запуска without-skill для eval 1 произвели пустой вывод»

### Шаг 6: Запишите заметки

Сохраните заметки в `{output_path}` как JSON массив строк:

```json
[
  "Утверждение 'Вывод является PDF файлом' проходит 100% в обеих конфигурациях — может не дифференцировать ценность скилла",
  "Eval 3 показывает высокую вариативность (50% ± 40%) — запуск 2 имел необычный провал",
  "Запуски without_skill последовательно проваливаются на ожиданиях извлечения таблиц",
  "Скилл добавляет 13с среднего времени выполнения, но улучшает pass rate на 50%"
]
```

## Руководства

**ДЕЛАЙТЕ:**
- Сообщайте, что вы наблюдаете в данных
- Будьте конкретны о том, к каким evals, ожиданиям или запускам вы относитесь
- Отмечайте паттерны, которые агрегированные метрики скрывали бы
- Предоставляйте контекст, который помогает интерпретировать числа

**НЕ ДЕЛАЙТЕ:**
- Предлагайте улучшения скилла (это для шага улучшения, а не бенчмаркинга)
- Делайте субъективные суждения о качестве («вывод был хорошим/плохим»)
- Спекулируйте о причинах без доказательств
- Повторяйте информацию, уже имеющуюся в сводках run_summary
